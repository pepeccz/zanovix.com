## Project Standards (compact rules — auto-injectable)

### Branding / brand canon (when touching design, copy, color, type, logo)

- **Bicroma estricta**: `--paper #F4F1EA` + `--ink #1F2A26` + **`--teal #3BAA8C`** como tercer color exclusivo. NO terra, NO mix con otros acentos.
- **Tipografía**: Newsreader serif (titulares y body), Inter sans (UI), JetBrains Mono (técnico). Pesos 400 o 500, **NUNCA bold (700+)**.
- **Sin italic** como mecanismo de énfasis editorial. Énfasis via teal + underline drawn o mono `//` prefix.
- **Isotipo** = flor de 8 pétalos. Es brand-stamp (header/footer/favicon), NO motion-language inline en body.
- **Voz**: cercana, joven, anti-hype, directa. NO emojis en formal, NO exclamaciones en titulares.
- **Tagline target**: "IA aplicada a empresas" — NO "ingeniería pausada con IA", NO "españolas".
- Canon: `~/zanovix-os/10-branding/DESIGN.md`. Tokens: `~/zanovix-os/10-branding/colores/tokens.css`.

### SEO/AEO (cuando se toque metadata, og, schema, sitemap, robots, llms.txt)

- Skill: `seo-aeo-best-practices` — JSON-LD, OG, hreflang, EEAT, robots.txt, sitemap.xml, llms.txt para AI search.
- Aplica en fase 4 (SEO/GEO/SEM) y de manera transversal cuando se cree cualquier ruta nueva.

### PR / commits / chained PRs

- **work-unit-commits**: commits por unidad de entrega, no por tipo de archivo. Tests y docs viajan junto al código.
- **chained-pr**: si una PR supera 400 líneas cambiadas, partir en chained PRs.
- **branch-pr**: workflow de creación de PR.
- **issue-creation**: issue antes que PR cuando hay scope ambiguo.

### Cognitive doc design (cuando se escriba README, RFC, arquitectura, onboarding)

### Comments (PR, issue, review, async)

- Skill: `comment-writer` — tono cálido directo humano, sin fórmulas vacías.

### Frontend design (cuando se construyan componentes, páginas, layouts)
